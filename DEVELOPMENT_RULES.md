# 광주광역사고조사센터 대시보드 - 핵심 아키텍처 및 시스템 규칙 (System Protocol)

본 문서는 대시보드 시스템의 **메모리(데이터), 디자인(UI/UX), 구성(아키텍처), 로직(알고리즘)** 영역에서 치열한 최적화를 통해 확립된 절대 규칙(Golden Rules)을 기록합니다. 향후 어떠한 업데이트나 AI 에이전트의 개입이 있더라도, 아래 규칙들을 기반으로 시스템의 안정성과 컨텍스트를 유지해야 합니다.

---

## 1. 메모리 및 데이터 저장 규칙 (Memory & Storage)

### 1.1 브라우저 저장소(Local Storage) 한도 방어
- **상황:** 공공데이터 `getNews_api02`를 1,000건씩 한 번에 불러올 경우 원본 HTML 태그가 포함되어 약 11.3MB의 거대한 용량을 차지하게 됨. 이는 브라우저의 Storage Quota(최대 5MB)를 초과하여 앱을 치명적인 크래시(QuotaExceededError)로 몰고 감.
- **규칙:** 캐시에 데이터를 저장하기 전 반드시 데이터 다이어트 작업(Mapping)을 거친다.
  - `contents` 등의 필드에서 `replace(/<[^>]*>?/gm, ' ')` 정규식을 수행하여 HTML 태그를 완벽히 제거하고 순수 텍스트(0.4MB 수준)로 압축한 뒤 저장한다.
- **쓰레기 데이터 청소 (Garbage Collection):** 데이터 저장 전, `localStorage.removeItem('public_accident_data')` 등 과거에 사용되었던 구버전 스토리지 키(v1, v2 등)를 명시적으로 파기하여 용량을 선제적으로 확보해야 한다.
- **안전장치 (Try-Catch):** 어떤 상황(시크릿 모드, 기타 브라우저 환경)에서도 스토리지 저장이 실패할 수 있으므로, 반드시 캐시 저장은 `try-catch` 구문으로 감싸 UI 렌더링에 영향을 미치지 않도록 방어 로직을 작성한다.

## 2. 디자인 및 UI/UX 원칙 (Design & UI/UX)

### 2.1 디스바이스 맞춤형 그래프 (Responsive Chart Configuration)
- 모바일(화면 너비 768px 미만) 환경에서는 통계 차트의 `options.indexAxis`를 `'y'`로 동적 변환(수평/가로형 막대)하여 좁은 화면에서 라벨(지역명)이 구겨지거나 겹치는 현상을 원천 차단한다. 데스크톱(PC)에서는 `'x'`축을 유지하여 시원한 뷰를 제공한다.

### 2.2 미니멀리즘과 뷰 계층도 (Visual Hierarchy)
- **정보의 우선순위:** '지역별 중대재해 비교' 그래프 섹션의 중요성이 높으므로, 일반 센터 운영 현황표보다 위에 배치하여 시각적 스크롤 동선의 우위를 점하게 한다.
- **불필요한 텍스트 배제:** "검색 결과 N건", "방금 전 업데이트됨" 등 화면을 지저분하게 만드는 내부 시스템 알림 문구나 아이콘은 과감히 제거하여 프리미엄(Premium)스럽고 정돈된 환경을 제공한다.
- **모달 애니메이션:** 상세 뉴스 모달 출력 시 뒷배경(body) 스크롤을 막고(`overflow: hidden`), CSS transition을 활용하여 부드러운 앱 사용 경험을 최우선시한다.

## 3. 구성 및 아키텍처 (Composition)

### 3.1 동기화 파이프라인 (Data Pipeline Sequence)
1. 시스템 초기화 직후 `await fetchPublicAccidentNews()`를 통해 즉시 캐시(v3)를 스캔한다.
2. 캐시가 있다면 API 통신을 단 한 건도 수행하지 않고 0.1초 내로 `processLoadedData()`를 호출해 렌더링을 끝마친다. (Zero-Delay)
3. 만약 캐시가 유통기한을 만료했거나 데이터가 없다면 그 즉시 `fallback`으로 Fetch API를 찔러 신규 데이터를 가져온 후 캐시에 저장(setItem)한다.
4. 이후 데이터 배열(`allPublicItems`)은 읽기 전용 상태로 남기고, 필터링 작업 시엔 `filteredPublicItems` 객체로 분리/가공하여 메모리 내에서 불변성(Immutability)을 유지하며 리렌더링한다.

## 4. 핵심 로직과 알고리즘 (Core Logic)

### 4.1 오전 9시 스마트 갱신 룰 (Dynamic 9 AM TTL)
- "4시간 단위"와 같은 막연한 주기를 철폐한다.
- **규칙:** 로컬 캐시에 저장된 시점이 **가장 최근에 지난 오전 9시 정각** 이전의 데이터라면 무조건 파기하고 새 데이터를 받는다.
  - *예시: 13일 오후 10시에 받은 데이터는 14일 오전 8시까지는 유효하지만, 14일 오전 9시 1분이 되는 순간 만료 처리됨.* 이를 통해 매일 아침 업무 시작과 동시에 자연스럽게 데이터가 클렌징(Cleansing) 되도록 스케줄링 로직을 구성했다.

### 4.2 중복 차트 파괴 로직 (Canvas Error Prevention)
- 캐시 갱신이나 필터링 버튼 클릭으로 인해 화면 정보가 다시 구성될 때, 자바스크립트는 기존 캔버스 위에 `new Chart()`를 올리면 충돌(Canvas is already in use 에러)을 일으킨다.
- **규칙:** 새 차트를 할당하기 전, `Chart.getChart(ctx)`를 검사하여 `existingChart.destroy()`를 호출하는 로직이 차트 그리기 함수의 첫 번째 관문으로 100% 필수 포함되어야 한다.

### 4.3 정교화된 정규식 파싱 엔진 (Extraction Engine)
- 공공 API 데이터 1,000건은 각기 다른 부서 코드로 혼재되어 있어 제목(`keyword`)만으로 지역/업종을 구분하면 광주 등 핵심 지역 데이터가 다량 누락된다.
- **해결 로직:** `keyword`와 `contents` 필드를 병합한 String 위에서 특정 키워드를 포착하는 다중 정규식 렌즈 맵핑 로직(`extractors.region` 등)을 거쳐 파싱된 부가 데이터(`_parsed` 객체)를 원본 데이터 객체에 봉인한 뒤 가공/필터링에 사용하는 아키텍처를 도입했다.
